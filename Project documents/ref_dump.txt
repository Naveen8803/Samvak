[108] ABSTRACT:
[109] This paper addresses security concerns in Ciphertext-policy attribute-based encryption (CP-ABE) for secure cloud data sharing. It tackles the key escrow problem by introducing a two-party key issuing protocol, preventing compromise by the key authority or cloud service provider. Additionally, the scheme enhances attribute expressiveness through weighted attributes, extending from binary to arbitrary states, reducing access policy complexity. This not only improves data sharing efficiency in cloud computing but also lowers storage costs and encryption complexity for cipher texts. The proposed solution ensures both security and efficiency, making it suitable for practical cloud applications.
[110] keywords: secure data sharing, attribute-based encryption, removing escrow, cloud computing

[127] INTRODUCTION
[128] 1.1 Motivation:
[129] The increasing reliance on cloud computing for data storage and sharing necessitates robust security measures to safeguard sensitive information. Ciphertext-policy attribute-based encryption (CP-ABE) offers a promising solution, but concerns such as the key escrow problem persist. Addressing this issue is crucial to instill trust in cloud-based systems, ensuring that sensitive data remains confidential and accessible only to authorized parties.
[130] 1.2 Problem Statement:
[131] The key problem addressed is the vulnerability of CP-ABE in cloud data sharing, particularly the key escrow issue. This vulnerability poses a risk of compromise by the key authority or cloud service provider. Additionally, the existing attribute expressiveness is limited, hindering efficient data sharing. The paper addresses these challenges by introducing a two-party key issuing protocol and enhancing attribute expressiveness through weighted attributes, aiming to improve security, reduce access policy complexity, and enhance efficiency in cloud data sharing applications.
[132] 1.3 Objective of the project:
[133] The objective of this project to enhance the security features for unauthorized decryption and unauthorized attribute manipulations on data while maintain the features of security. Aiming to make the system more secure and practical for use.
[134] 1.4 Scope:
[135] This paper focuses on enhancing the security of Ciphertext-policy attribute-based encryption (CP-ABE) for cloud data sharing. It introduces a two-party key issuing protocol to mitigate the key escrow problem, ensuring security against compromises by the key authority or cloud service provider. The scheme also extends attribute expressiveness through weighted attributes, reducing access policy complexity and improving data sharing efficiency in cloud computing. The proposed 
[136] 
[137] solution aims to address security challenges, enhance attribute expressiveness, and optimize efficiency, making it applicable to practical cloud applications.
[138] 1.5 Project Introduction:
[139] In the rapidly evolving landscape of cloud computing, ensuring the security and privacy of shared data is paramount. Ciphertext-policy attribute-based encryption (CP-ABE) has emerged as a promising cryptographic technique for secure data sharing in the cloud. However, the persistent challenge of the key escrow problem threatens the integrity of CP-ABE systems, raising concerns about the confidentiality of sensitive information. This paper presents a novel approach to address these security concerns and enhance the efficiency of data sharing in cloud environments.
[140] The key contribution of this work lies in the introduction of a two-party key issuing protocol, which effectively mitigates the key escrow problem inherent in CP-ABE systems. By distributing key issuance authority between two parties, namely the key authority and the cloud service provider, the protocol prevents any single entity from compromising the security of the system. This innovative solution ensures that sensitive data remains confidential and accessible only to authorized users, bolstering trust in cloud-based data sharing mechanisms.
[141] Furthermore, the proposed scheme enhances attribute expressiveness through the incorporation of weighted attributes, allowing for finer granularity in access control policies. By extending attribute states from binary to arbitrary values, the complexity of access policies is reduced, thereby improving the efficiency of data sharing operations. This enhancement not only streamlines the process of data access management but also contributes to lower storage costs and reduced encryption complexity for ciphertexts.
[142] In summary, the proposed solution offers a comprehensive approach to addressing security concerns in CP-ABE for secure cloud data sharing. By removing the key escrow problem, enhancing attribute expressiveness, and improving efficiency, this solution holds significant promise for practical applications in cloud computing environments.

[205] 3. SYSTEM ANALYSIS
[206] 3.1 Existing System
[207] The existing system employs Ciphertext-policy attribute-based encryption (CP-ABE) for secure cloud data sharing. However, it faces several challenges that need to be addressed. The key escrow problem is a significant concern, as the key authority or cloud service provider could compromise the security of the system. Additionally, the attribute expressiveness in the access policies is limited to binary states, which can lead to increased complexity in defining access policies. This can affect data sharing efficiency, raise storage costs, and complicate the encryption process for cipher texts.
[208] 3.2 Disadvantages
[209] 1. Key Escrow Vulnerability: The existing system is susceptible to the key escrow problem, posing a risk of compromise by the key authority or cloud service provider.
[210] 2. Limited Attribute Expressiveness: The binary attribute states in the access policies restrict the system's ability to efficiently represent complex access requirements.
[211] 3. Increased Access Policy Complexity: The binary attribute model may result in higher complexity when defining access policies, potentially affecting the usability of the system.
[212] 4. Higher Storage Costs: The limited attribute expressiveness and key escrow concerns may contribute to higher storage costs for managing encryption keys and access policies.
[213] 5. Encryption Complexity: The existing system's reliance on binary attribute states may lead to increased encryption complexity for cipher texts, impacting overall system efficiency.	
[214] 3.3 Proposed System
[215] The proposed system introduces a two-party key issuing protocol to address the key escrow problem and enhance security. It also extends attribute expressiveness by introducing weighted attributes, allowing for arbitrary attribute states. This extension aims to reduce access policy complexity, improving data sharing efficiency in cloud computing. Additionally, the proposed system aims to lower storage costs and simplify encryption complexity for cipher texts.
[216] 3.4 Advantages
[217] 1. Key Escrow Mitigation: The two-party key issuing protocol effectively mitigates the key escrow problem, enhancing the overall security of the system.
[218] 
[219] 2.Enhanced Attribute Expressiveness: The introduction of weighted attributes allows for arbitrary 
[220] 
[221] attribute states, reducing access policy complexity and increasing flexibility in defining access requirements.
[222] 3. Improved Data Sharing Efficiency: The proposed system is designed to enhance data sharing efficiency in cloud computing, leading to quicker and more streamlined sharing processes.
[223] 4.Lower Storage Costs: By addressing key escrow concerns and improving attribute expressiveness, the proposed system aims to reduce overall storage costs associated with encryption keys and access policies.
[224] 5.Simplified Encryption Complexity: The extension from binary to arbitrary attribute states contributes to simplified encryption processes for cipher texts, promoting efficiency in the encryption and decryption operations.

[232] 4.1 Functional and non-functional requirements
[233] Requirement’s analysis is very critical process that enables the success of a system or software project to be assessed. Requirements are generally split into two types: Functional and non-functional requirements.
[234] Functional Requirements: These are the requirements that the end user specifically demands as basic facilities that the system should offer. All these functionalities need to be necessarily incorporated into the system as a part of the contract. These are represented or stated in the form of input to be given to the system, the operation performed and the output expected. They are basically the requirements stated by the user which one can see directly in the final product, unlike the non-functional requirements.
[235] Examples of functional requirements: 
[236] Authentication of user whenever he/she logs into the system
[237] System shutdown in case of a cyber-attack
[238] A verification email is sent to user whenever he/she register for the first time on some software system.
[239] Non-functional requirements: These are basically the quality constraints that the system must satisfy according to the project contract. The priority or extent to which these factors are implemented varies from one project to other. They are also called non-behavioral requirements.
They basically deal with issues like:
[240] Portability
[241] Security
[242] Maintainability
[243] Reliability
[244] Scalability
[245] Performance
[246] Reusability
[247] Flexibility

[250] Emails should be sent with a latency of no greater than 12 hours from such an activity.

[1342] 7.1 Feasibility Study
[1343] The feasibility of the project is analysed in this phase and business proposal is put forth with a very general plan for the project and some cost estimates. During system analysis the feasibility study of the proposed system is to be carried out. This is to ensure that the proposed system is not a burden to the company.  For feasibility analysis, some understanding of the major requirements for the system is essential.
[1344] Three key considerations involved in the feasibility analysis are
[1345] Economical feasibility
[1346] Technical feasibility
[1347] Social feasibility
[1348] Economical Feasibility
[1349] This study is carried out to check the economic impact that the system will have on the organization. The amount of fund that the company can pour into the research and development of the system is limited. The expenditures must be justified. Thus the developed system as well within the budget and this was achieved because most of the technologies used are freely available. Only the customized products had to be purchased.
[1350] Technical Feasibility
[1351] This study is carried out to check the technical feasibility, that is, the technical requirements of the system. Any system developed must not have a high demand on the available technical resources. This will lead to high demands on the available technical resources. This will lead to high demands being placed on the client. The developed system must have a modest requirement, as only minimal or null changes are required for implementing this system.
[1352] Social Feasibility
[1353] 
[1354] The aspect of study is to check the level of acceptance of the system by the user. This includes the process of training the user to use the system efficiently. The user must not feel threatened by the system, instead must accept it as a necessity. The level of acceptance by the users solely depends on the methods that are employed to educate the user about the system and to make him familiar with it. His level of confidence must be raised so that he is also able to make some constructive criticism, which is welcomed, as he is the final user of the system.

[1356] System Testing
[1357] The purpose of testing is to discover errors. Testing is the process of trying to discover every conceivable fault or weakness in a work product. It provides a way to check the functionality of components, sub-assemblies, assemblies and/or a finished product It is the process of exercising software with the intent of ensuring that the
[1358] Software system meets its requirements and user expectations and does not fail in an unacceptable manner. There are various types of test. Each test type addresses a specific testing requirement.
[1359] 
[1360] 7.2 Types of Tests
[1361] 7.2.1 Unit testing
[1362] Unit testing involves the design of test cases that validate that the internal program logic is functioning properly, and that program inputs produce valid outputs. All decision branches and internal code flow should be validated. It is the testing of individual software units of the application .it is done after the completion of an individual unit before integration. This is a structural testing, that relies on knowledge of its construction and is invasive. Unit tests perform basic tests at component level and test a specific business process, application, and/or system configuration. Unit tests ensure that each unique path of a business process performs accurately to the documented specifications and contains clearly defined inputs and expected results.
[1363] 7.2.2 Integration testing
[1364] Integration tests are designed to test integrated software components to determine if they actually run as one program.  Testing is event driven and is more concerned with the basic outcome of screens 
[1365] 
[1366] or fields. Integration tests demonstrate that although the components were individually satisfaction, as shown by successfully unit testing, the combination of components is correct and consistent. Integration testing is specifically aimed at   exposing the problems that arise from the combination of components.
[1367] Software integration testing is the incremental integration testing of two or more integrated software components on a single platform to produce failures caused by interface defects.
[1368] The task of the integration test is to check that components or software applications, e.g. components in a software system or – one step up – software applications at the company level – interact without error.
